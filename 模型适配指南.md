# 模型推理适配指南

> **注意**：提交 PR 前务必阅读文末 [第 7 节 CI 说明](#7-提交-pr)，确保 run.sh 满足所有 CI 要求，否则 PR 将无法通过。

本文档介绍如何将推理模型适配至太初 AI 加速卡（SDAA 平台），使其能够基于 TecovLLM 框架正常运行。整体流程如下：

| # | 阶段 | 说明 |
|---|------|------|
| 1 | 检查推理环境 | 确认开发环境（硬件 + 容器）满足任务需求 |
| 2 | 模型适配 | 对模型源码和推理接口进行适配，使其能在 TecovLLM 框架中运行 |
| 3 | 自定义算子开发与接入 | 为 SDAA 暂不支持或精度不达标的算子开发适配实现 |
| 4 | 精度分析 | 验证推理精度是否达标，若不达标则进行调试 |
| 5 | 性能分析 | 测试吞吐、延迟等关键指标，若不达标则进行调优 |
| 6 | 添加 README | 为适配的模型编写推理使用说明 |
| 7 | 提交 PR | 完成所有测试后，将代码提交至仓库 |

## 1. 检查推理环境

开始适配前，建议先熟悉开发环境的软硬件配置，确保容器、框架、加速卡等资源就绪，以便适配和推理过程顺利进行。

### 1.1 检查硬件信息

使用 `teco-smi` 命令查看太初加速卡的设备列表与工作状态：

```bash
teco-smi
```

输出示例：

```
+-----------------------------------------------------------------------------+
|  TECO-SMI: 1.15.0        SDAADriver: 3.1.0        SDAARuntime: 3.1.0        |
|-------------------------------+----------------------+----------------------|
| Index  Name                   | Bus-Id               | Health      SPE-Util |
|        Temp          Pwr Usage|          Memory-Usage|                      |
|=============================================================================|
|   0    TECO_AICARD_01         | 00000002:05:00.0     | OK                0% |
|        42C                98W |        0MB / 65536MB |                      |
|-------------------------------+----------------------+----------------------|
|   1    TECO_AICARD_01         | 00000002:06:00.0     | OK                0% |
|        40C                92W |        0MB / 65536MB |                      |
+-------------------------------+----------------------+----------------------+
```

重点关注以下字段：

| 字段 | 说明 | 正常状态 | 异常处理 |
|------|------|----------|----------|
| **Memory-Usage** | 设备内存使用状态（已用 / 总量） | 已用内存远小于总量 | 内存占用异常时检查残留进程 |
| **Health** | 设备健康状态 | `OK` | 出现 `DEVICE_LOST`、`HEARTBEAT_ERROR` 等异常时，请联系太初技术支持 |
| **SPE-Util** | SPE 计算核心使用率 | 正常数值 | 显示 `N/A` 表示芯片掉卡，请联系太初技术支持 |

**步骤 1：查看 vLLM / TecovLLM 版本**

```bash

```

若终端成功输出 Torch-SDAA 及各依赖组件的版本信息，表明运行环境就绪。示例如下：

## 2. 模型适配

### 2.1 模型分析

适配前请完成以下确认，以保障后续流程顺利开展：

1. **模型可运行验证**：确认模型在 CPU 或 GPU 环境下能正常加载并完成基础推理，无报错或异常退出。
2. **量化格式确认**：当前仅支持 **FP16** 和 **INT8** 两种量化格式，其他格式暂不兼容。
3. **并行策略确认**：
   - **TP（张量并行）**：按硬件核组数配置，单卡为 4 核组，单卡运行时 TP = 4。
   - **PP / DP / EP**：并行规则与标准 CUDA 保持一致。

### 2.2 模型组网

使用 `torch` 和 `torch_sdaa` 提供的算子与函数实现模型的网络结构。具体实现方式请参考 vLLM 官网的 [OPT 模型适配示例](https://docs.vllm.ai/en/stable/contributing/model/basic/)。

本文使用的 Qwen3-0.6B 模型网络结构实现代码，见 [qwen3_dense.py](../models/Qwen3-0.6B/qwen3_dense.py)。

### 2.3 模型注册

TecovLLM 通过插件机制 `tecovllm_models` 管理自定义模型。该插件已集成在 TecovLLM-ModelZoo 仓库中，目录结构如下：

```
tecovllm_models/
├── __init__.py              # 模型注册入口
├── layers/                  # 公共模块（如 RMSNorm、RoPE 等）
│   └── ...
└── models/                  # 模型组网
    ├── __init__.py
    └── ...
```

注册流程分为以下三步：

**步骤 1：放置模型组网文件**

将实现好的模型组网文件放到 `tecovllm_models/models/` 目录下。例如，将 `qwen3_dense.py` 放入该目录：

```
tecovllm_models/models/
├── __init__.py
└── qwen3_dense.py         # 新增的模型组网文件
```

模型组网文件中需实现 `embed_input_ids`、`forward`、`compute_logits`、`load_weights` 等接口，具体实现请参考已有示例。

**步骤 2：注册模型**

在 `tecovllm_models/__init__.py` 中添加 `ModelRegistry.register_model` 调用：

```python
from vllm import ModelRegistry

ModelRegistry.register_model(
    model_arch="Qwen3DenseForCausalLM",
    model_cls="tecovllm_models.models.qwen3_dense:Qwen3DenseForCausalLM"
)
```

其中：

| 参数 | 说明 |
|------|------|
| `model_arch` | 模型架构名，**必须与模型配置文件中 `architectures[0]` 一致**。若不一致，启动时需通过 `--hf-override '{"architectures": ["Qwen3DenseForCausalLM"]}'` 覆盖 |
| `model_cls` | 模型类的模块导入路径，格式为 `模块路径:类名` |

> **一次注册多个模型**：如需同时注册多个模型，直接在后面追加 `ModelRegistry.register_model(...)` 即可，无需修改其他代码。

**步骤 3：编译并安装插件**

在 tecovllm_models 目录下运行：

```bash
python setup.py bdist_wheel
```

或使用开发模式免去每次修改后重复编译：

```bash
python setup.py develop
```

编译安装完成后，模型即可被 TecovLLM 推理引擎识别并正常加载运行。

### 2.4 运行模型

#### 2.4.1 离线推理

参考 [离线推理文档](http://docs.tecorigin.com/release/teco_vllm/v3.1.0/#d6692347dc1c533ab3651f83213ae646)，完成模型离线推理验证。

#### 2.4.2 在线推理

参考 [在线推理文档](http://docs.tecorigin.com/release/teco_vllm/v3.1.0/#43351f4b9e305913b221a171b0a5a59d)，完成模型在线推理验证。

## 3. 自定义算子开发与接入

在模型迁移过程中，开发者通常需要在以下场景中开发自定义算子：
- SDAA 硬件暂不支持该算子
- 需要对特定算子进行**性能优化**（如算子融合、优化特定 shape 下的算子性能等）

此时可通过定制开发适配算子并将其接入 TecovLLM 推理框架来解决。太初官方在 [算子开发仓库](https://github.com/Tecorigin/teco-ops) 中提供了部分基础版算子供开发者作为参考示例。算子开发及 PyTorch 框架接入方法，请参考该仓库中的相关文档。

## 4. 精度分析

模型适配完成后，需通过精度验证确认推理结果正确性。具体流程如下：

### 4.1 基准精度（CUDA 环境）

**环境要求**：vLLM 版本为 **0.20.2**，Evalscope 版本为 **1.8.1**。

**配置要求**：使用 vLLM **默认配置**运行，不启用任何优化策略（如量化、并行、PD 分离等）。

> **采样参数**：各模型的采样参数（temperature、top_p、top_k 等）均使用模型官方推荐的设置方式，以保证评测结果与官方基准可比。

以下为四个目标模型在 CUDA 环境下的基准精度运行命令：

**评测数据集**：wmt24pp（子集 en-zh_cn）

**启动 vLLM 服务**：

```bash

    --trust-remote-code \
    --tensor-parallel-size 1 \
    --port <port> \

    --max_model_len 262144 \
    --host <ip>
```

#### 基准精度汇总

下表汇总了四个模型在 CUDA 环境下的基准精度评测结果。每个数据集独立运行 **10 轮**，得分以 **平均值 ± 波动范围** 表示（波动范围 = (最大值 − 最小值) / 2，体现 10 轮测试的分数波动）。

| 模型 | 数据集 | 子集 | Case 数 | 评分指标 | 得分 |
|------|--------|------|---------|----------|------|
| Hy-MT2-1.8B | wmt24pp | en-zh_cn | 951 | mean_bert_score | 0.9405 ± 0.0003 |
| MiniCPM5-1B | mmlu_pro | computer science | 410 | mean_acc | 0.4688 ± 0.0342 |
| InternVL3_5-8B | mmlu_pro | computer science | 410 | mean_acc | 0.6910 ± 0.0281 |
| InternVL3_5-8B | mmmu_pro | Accounting | 58 | mean_acc | 0.4862 ± 0.0776 |
| gemma-4-12B-it | mmlu_pro | computer science | 410 | mean_acc | 0.8458 ± 0.0098 |
| gemma-4-12B-it | mmmu_pro | Accounting | 58 | mean_acc | 0.7500 ± 0.0776 |

> **说明**：mmmu_pro 数据集 case 数较少（仅 58 条），单题得分波动对整体指标影响较大，因此该数据集的 ± 波动范围相对更宽。

### 4.2 目标精度（SDAA 环境）

在 SDAA 环境上运行评测时，需遵循以下规则：

- **vLLM 服务启动命令**：可**按需调整**，以达到 SDAA 环境下的最优推理性能。
- **Evalscope 数据集评测命令**：须与 CUDA 环境**完全一致**（包括数据集、子集、采样参数等），确保评测对比条件统一。

### 4.3 精度判定标准

以 4.1 节 CUDA 基准精度得分作为基线，各数据集在 SDAA 环境下的可接受波动范围如下：

| 数据集 | 可接受范围 |
|--------|-----------|
| wmt24pp | CUDA 基线 **+0.001** |
| mmlu_pro | CUDA 基线 **±0.05** |
| mmmu_pro | CUDA 基线 **±0.1** |

各模型按数据集对照上表，SDAA 得分落入允许区间即视为精度达标。

### 4.4 精度分析工具

若精度不达标，可使用精度分析工具定位差异来源：

| 工具 | 说明 | 参考文档 |
|------|------|----------|
| **精度分析工具** | 当模型精度异常时，使用该工具逐层对齐 CUDA 与 SDAA 的推理精度，定位精度异常的层，帮助加快精度排查进度 | [精度分析工具使用指南](http://docs.tecorigin.net/release/teco_precision/) |
| **Evalscope** | 评测工具，用于在多任务场景下验证模型整体精度表现 | [Evalscope 模型评测指南](https://evalscope.readthedocs.io/zh-cn/latest/get_started/supported_dataset/index.html) |

## 5. 性能分析

性能测试使用 **Evalscope** 工具，**vLLM 服务启动命令须与精度测试（第 4 节）完全一致**，直接复用已启动的服务，无需单独启动。

**Warmup**：正式测试前，先发送一次 warmup 请求（bs=1, 输入 1024 tokens + 输出 200 tokens），预热推理引擎，避免首次请求的冷启动开销影响测试结果。

### 5.1 文本类性能测试（所有模型）

所有模型均需完成以下文本场景的性能测试，取 **TTFT**（Time To First Token，首 token 延迟）和 **TPOT**（Time Per Output Token，每输出 token 耗时）作为评测指标。

| 序号 | 场景 | 并发数 | 请求数 | 输入长度 | 输出长度 |
|------|------|--------|--------|----------|----------|
| T1 | 长文本串行 | 1 | 20 | 65536 (64K) | 200 |
| T2 | 超长文本串行 | 1 | 20 | 122880 (120K) | 200 |
| T3 | 短文本高并发 | 32 | 640 | 1024 (1K) | 200 |

**T1 — 长文本串行 (bs=1, 64K→200)**：

```bash
evalscope perf \
    --model <model_name> \
    --api-url http://<ip>:<port>/v1/chat/completions \
    --api openai \
    --tokenizer-path <tokenizer_path> \
    --dataset random \
    --min-prompt-length 65536 \
    --max-prompt-length 65536 \
    --max-tokens 200 \
    --min-tokens 200 \
    --extra-args '{"ignore_eos": true}' \
    --parallel 1 \
    --number 20 \
    --stream
```

**T2 — 超长文本串行 (bs=1, 120K→200)**：

```bash
evalscope perf \
    --model <model_name> \
    --api-url http://<ip>:<port>/v1/chat/completions \
    --api openai \
    --tokenizer-path <tokenizer_path> \
    --dataset random \
    --min-prompt-length 122880 \
    --max-prompt-length 122880 \
    --max-tokens 200 \
    --min-tokens 200 \
    --extra-args '{"ignore_eos": true}' \
    --parallel 1 \
    --number 20 \
    --stream
```

**T3 — 短文本高并发 (bs=32, 1K→200)**：

```bash
evalscope perf \
    --model <model_name> \
    --api-url http://<ip>:<port>/v1/chat/completions \
    --api openai \
    --tokenizer-path <tokenizer_path> \
```

### 5.3 命令参数说明

| 参数 | 说明 |
|------|------|
| `--model` | 模型名称或路径 |
| `--api-url` | vLLM 服务 OpenAI 兼容 API 地址 |
| `--api` | API 类型，固定为 `openai` |
| `--tokenizer-path` | tokenizer 路径，用于计算 token 数 |
| `--dataset` | 数据集类型：`random`（随机文本）、`random_vl`（随机图片） |
| `--min-prompt-length` / `--max-prompt-length` | 输入 prompt 的 token 数范围，两端设为相同值即固定长度 |
| `--min-tokens` / `--max-tokens` | 输出 token 数范围，两端设为相同值即固定长度 |
| `--image-width` / `--image-height` | 随机图片的宽/高（像素） |
| `--image-num` | 每次请求携带的图片数量 |
| `--image-format` | 图片颜色格式，固定为 `RGB` |
| `--extra-args` | 透传给 API 的额外参数，`ignore_eos: true` 使模型忽略 EOS 持续生成至指定 token 数 |
| `--parallel` | 并发请求数（对应 batch size） |
| `--number` | 总请求数（= bs × 20） |
| `--stream` | 启用流式输出，按 token 粒度统计时延 |

## 7. 提交 PR

完成所有测试并通过后，将代码提交到 TecovLLM-ModelZoo 仓库。PR 须包含以下三个部分：

1. **模型适配代码**：模型组网文件、注册代码等，按仓库现有结构放置。
2. **`model-adaptations/<团队名称>/README.md`**：适配模型的推理使用说明，格式参考 [第 6 节](#6-添加-readme)。
3. **`model-adaptations/<团队名称>/run.sh`**：可直接运行的 vLLM 服务启动命令，包含完整的模型路径、端口及所有必要参数。

PR 提交流程请参考 [PR 提交规范](./pr提交规范.md)。


> **CI 要求**：
> - **`tools/ci_pipline/` 下的文件不可修改**，否则 PR 无法通过检查、无法合并。
> - **`model-adaptations/<团队名称>/run.sh`** 中的命令**必须包含 `--served-model-name`**，且参数必须为 `"Hy-MT2-1.8B"`、`"MiniCPM5-1B"`、`"InternVL3_5-8B"`、`"gemma-4-12B-it"` 四个之一，否则 CI 会挂。
> - **vllm serve 请勿使用默认端口**，必须通过 `--port` 显式指定端口号，避免与其他任务端口冲突。
> - **run.sh 中必须包含 `--trust-remote-code`**，否则部分模型的 tokenizer 或模型配置可能无法正常加载。
